# LobsterPool Roadmap (as of 2026-03-09)

## 1. 项目现状

截至 2026 年 3 月 9 日，LobsterPool 已经不是概念验证，而是一个具备基础可用性的 OpenClaw 实例开通平台。

当前已落地能力：

- 用户注册、登录、修改密码
- 管理员总览、用户实例额度管理、模板创建
- 模板选择、多集群目标选择、实例创建 / 查看 / 删除
- Mock Provider 与 Kubernetes Provider 双实现
- SQLite 持久化、Docker 构建、Kubernetes 部署清单
- GitHub Actions 基础 CI（后端测试、前端 lint/build）
- Playwright E2E 覆盖关键主链路

已在仓库内验证的基线：

- `make test-backend` 通过
- `make lint-frontend` 通过
- `make build` 通过
- `make test-e2e` 通过（2 条用例）

## 2. 当前判断

这个项目当前处于“功能主链路可用，但距离生产级平台还有一段工程化距离”的阶段。

强项：

- 核心业务链路完整，前后端闭环已经跑通
- Provider 抽象清晰，Mock / K8s 切换成本低
- 多集群能力已经进入主干，不需要重做架构
- 基础测试和 CI 已经具备，说明项目可持续迭代

主要短板：

- 安全基线还不够强：默认管理员初始化、JWT 本地存储、缺少登录限流 / 审计 / 密码策略
- 编排可靠性不足：实例状态主要依赖请求时刷新，没有后台 reconciliation / drift 修复
- 平台运营能力偏弱：缺少集群健康、容量、失败原因、审计日志、成本与配额视图
- 交付能力仍偏基础：CI 还没有覆盖 E2E、镜像发布、发布版本管理和回滚流程
- 管理能力还不完整：模板只有创建，没有生命周期治理；用户治理目前只有额度修改

## 3. 2026 Roadmap

### Phase 1: 生产基线补齐（2026-03-15 至 2026-04-30）

目标：把“能跑”提升到“可以小规模稳定上线”。

重点交付：

- 去掉生产环境默认 `admin/admin` 依赖，改为一次性 bootstrap 或显式初始化流程
- 补登录限流、密码复杂度校验、登录失败审计、关键管理操作审计
- 梳理认证方案，至少完成 JWT 生命周期治理；更优方案是切到 HttpOnly Cookie
- 补充配置校验与启动失败提示，避免集群配置错误时静默带病启动
- 增加运维文档：部署步骤、升级步骤、备份恢复、常见故障处理

完成标准：

- 新环境默认不暴露弱口令
- 管理端关键操作有审计记录
- 认证失败、配置错误、权限错误都能被清晰定位

### Phase 2: 编排可靠性建设（2026-05-01 至 2026-06-30）

目标：把实例生命周期从“同步调用”升级为“可恢复、可追踪、可收敛”。

重点交付：

- 增加实例状态同步任务，定期回写真实 K8s 状态
- 为创建 / 删除流程补偿失败场景，减少数据库与 K8s 资源状态不一致
- 引入 provider 层单元测试，覆盖 Secret / Deployment / Service 资源生成与异常路径
- 在实例详情和管理员视图中展示更细的失败原因与最近状态更新时间
- 为多集群场景补 cluster health 检查与不可用集群降级策略

完成标准：

- 实例状态不再依赖用户手动刷新详情页才更新
- 资源创建 / 删除失败能够自动补偿或明确暴露待处理状态
- Provider 成为有测试保护的稳定边界

### Phase 3: 平台治理与运营（2026-07-01 至 2026-09-30）

目标：把 LobsterPool 从“实例开通工具”升级为“可运营的平台”。

重点交付：

- 模板生命周期管理：编辑、下线、版本治理、默认版本切换
- 用户治理能力：角色管理、禁用用户、按用户组 / 集群设置额度
- 集群运营面板：容量、实例分布、失败率、近 7 天创建趋势
- 增加筛选、搜索、排序、分页，提升管理员处理效率
- 为实例引入标签 / 备注 / 所属团队字段，支撑后续成本归集

完成标准：

- 管理员可以完整管理模板与用户，而不只是“新增”和“改额度”
- 可以明确回答“谁在用、用在哪个集群、失败在哪里、占用了多少资源”

### Phase 4: 企业化交付（2026-10-01 至 2026-12-31）

目标：让项目具备更正式的企业内部推广条件。

重点交付：

- 增加 SSO / OIDC 登录，替代纯本地账号体系
- 完善 CI/CD：纳入 E2E、镜像构建发布、版本号管理、发布检查
- 提供 Helm Chart 或更完整的生产部署包
- 增加监控指标、结构化日志、告警规则、数据库备份策略
- 形成 release note、升级兼容策略和运维值班手册

完成标准：

- 新版本可标准化发布、回滚、观测
- 身份体系与企业账号体系对接
- 平台具备跨团队推广的最小运营保障

## 4. 优先级原则

2026 年不建议同时铺太多产品需求。优先顺序应固定为：

1. 安全与认证
2. 生命周期可靠性
3. 管理与运营效率
4. 企业化集成

原因很直接：如果安全和状态收敛没做好，越快推广，后续运维成本越高。

## 5. 建议的里程碑指标

建议用下面几项指标判断 roadmap 是否兑现：

- 实例创建成功率
- 实例删除成功率
- 创建到 Running 的中位耗时
- 状态不一致实例数
- 登录失败率与异常登录告警数
- 每周活跃用户数 / 活跃实例数
- 多集群分布与集群不可用恢复时间

## 6. 明确不在当前阶段优先做的事

以下事项可以暂缓，避免 roadmap 失焦：

- 复杂计费系统
- 高度自定义审批流
- 插件市场
- 过早的多云抽象

先把“安全、稳定、可运维”做好，项目价值会比继续堆功能更快释放。
