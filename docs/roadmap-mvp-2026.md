# LobsterPool MVP Roadmap (focused version)

_Updated: 2026-03-09_

## 1. 目标重述

LobsterPool 的 MVP 目标不是先做成一个完整的平台，而是先证明下面这件事：

> 用户只需要提交必要凭证和模板选择，就能领养一只可运行的 OpenClaw，不需要理解底层 Kubernetes 部署细节。

因此，roadmap 的优先级应该围绕“用户是否真的成功领养并用起来”来排，而不是先围绕平台治理和工程完备度来排。

## 2. 当前现状

当前项目已经具备 MVP 的基础骨架：

- 用户注册、登录、改密
- 管理员后台、模板创建、实例额度管理
- 模板选择、实例创建、查看、删除
- Mock Provider 与 Kubernetes Provider
- SQLite 持久化
- 基础 CI
- E2E 已覆盖主链路

但当前仍存在一个核心问题：

- “实例创建成功”已经成立
- “用户真正能访问并使用实例”还没有完全产品化闭环

所以接下来不应该继续平均发力，而应该优先补全 MVP 主旅程。

## 3. MVP 主旅程

MVP 只聚焦下面这条主线：

1. 用户进入系统
2. 登录 / 注册
3. 选择模板
4. 填写 API Key 和 Mattermost Bot Token
5. 提交创建请求
6. 系统启动对应实例
7. 用户看到明确状态
8. 用户拿到真正可访问的入口
9. 用户可以删除实例并再次申请

这 9 步只要有一步不能稳定闭环，MVP 就还没有完成。

## 4. 新的优先级原则

新的优先级固定为：

1. 主旅程闭环
2. 管理员支撑 MVP 运行
3. 最小可上线基线
4. 安全 / 运维 / 排障增强

也就是说：

- 先解决“能不能领养并用起来”
- 再解决“管理员能不能支撑这个流程”
- 最后再补更重的安全与运维能力

## 5. Roadmap

### Phase 1: 主旅程闭环（最高优先级）

目标：让用户从“申请龙虾”到“打开龙虾”真正跑通。

重点交付：

- 创建完成后返回真实可访问地址，而不是仅返回集群内地址
- 明确区分实例状态：创建中、运行中、失败、删除中
- 在实例列表和详情页中突出“可用入口”而不是底层资源名
- 优化创建失败反馈，让用户知道是凭证、模板、额度还是系统问题
- 删除实例后给出明确回收结果，并允许用户继续创建

完成标准：

- 用户创建实例后，可以拿到一个自己真的能打开的入口
- 用户不需要理解 Deployment、Service、Namespace
- 用户能看懂当前是不是“已经可以用了”

### Phase 2: 管理员最小支撑能力

目标：让管理员可以维持 MVP 运转，而不只是“看见数据”。

重点交付：

- 模板管理补齐到可用状态：至少支持编辑和下线
- 管理员可查看并处理异常实例
- 管理员可查看用户额度与实例分布，并执行最基本治理动作
- 管理后台提供更清晰的异常聚合视图，而不是只展示静态总览

完成标准：

- 管理员不仅能“新增模板”，还能维护模板
- 管理员遇到异常实例时有处理手段
- 管理员能支撑小规模内部试运行

### Phase 3: 最小可上线基线

目标：把“可演示”提升到“可小范围使用”。

重点交付：

- 区分开发环境和生产环境的初始化行为
- 增加必要配置校验与启动失败提示
- 补部署、升级、备份恢复等最小运维文档
- 补最小认证和权限安全基线，但不抢占主旅程优先级

完成标准：

- 小范围内部使用时，不会因为配置或初始化问题频繁卡死
- 上线、升级、恢复流程有最基本说明
- 开发便利和生产默认行为被明确区分

### Phase 4: 安全、运维、排障增强

目标：在主旅程已经稳定后，提升长期可运营性。

重点交付：

- 登录限流、登录失败审计、关键管理操作审计
- JWT 生命周期治理或迁移到更稳妥的认证方案
- 实例状态后台同步与失败补偿
- 集群健康、容量、失败原因、趋势等运营视图
- 发布、回滚、监控告警、结构化日志

完成标准：

- 项目不只是“能用”，而是逐步具备更可靠的长期维护能力
- 但这些工作不再阻塞 MVP 主价值验证

## 6. 2026 年真正应该先做的任务

如果只列 MVP 阶段最关键的任务，建议顺序如下：

1. 把实例访问入口做成真实可用地址
2. 把实例状态和失败反馈做清楚
3. 补齐管理员对模板和异常实例的最小治理能力
4. 明确开发 / 生产初始化和部署路径
5. 再逐步补安全、运维、排障能力

## 7. 明确延后

下面这些不是当前 MVP 阶段的第一优先级：

- 完整审计体系
- 复杂权限模型
- 全量运维大盘
- 多云或过早抽象
- 高度企业化集成
- 复杂计费和审批流

这些并不是不重要，而是不应该排在“用户能不能真正用到龙虾”之前。

## 8. 一句话版本

LobsterPool 的下一阶段不该优先把自己做成一个重平台，而应该先把“领养一只龙虾并真正用起来”这件事做扎实。
